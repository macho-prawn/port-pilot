What it shows

ports is built for the workflow where the machine is clearly listening, but the raw socket tools do not tell you enough to act confidently. It resolves the owning process, working directory, runtime hints, framework/language guess, memory, uptime, and command summary in one pass.

When a field cannot be resolved, it renders as ------ so the table shape stays stable and easy to scan.

Stack

Runtime

Rust, clap, ratatui, crossterm, owo-colors

Discovery

`listeners` first, `lsof` fallback on Unix, host-level fallback when needed

Enrichment

sysinfo metadata, cwd lookup, memory, uptime, framework and language inference

Commands

ports

Open the live TUI with auto-refresh and interactive shortcuts.

ports ls

Print a static table of all listening ports.

ports check 3000

Show a detailed view for one port, or say that it is free.

ports kill 3000

Terminate every process listening on that port. Graceful first, then force.

ports --json ls

Emit machine-readable output for automation and scripts.

ports help kill

Show focused help for one command or switch.

Detection rules

ports uses deterministic heuristics rather than opaque guessing. It inspects command lines, executable names, and nearby project markers such as package.json, Cargo.toml, pyproject.toml, Gemfile, go.mod, and framework config files like next.config.* or vite.config.*.

On Linux, Docker, and containerd-backed environments, the tool also enriches fallback listeners when the runtime exposes enough metadata to tie the socket back to a process or published service.

Common workflows

# What is using port 3000?
ports check 3000
# Show everything listening right now
ports ls
# Kill a stuck dev server
ports kill 5173
# Open the live monitor
ports --interval 5
